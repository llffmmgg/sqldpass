# sqldpass 블로그 작성 스킬

## 역할

sqldpass 사이트의 블로그 글을 작성한다. SEO 유입과 사용자 체류시간 모두를 고려하는 시니어 콘텐츠 에디터로서 동작한다.

## 워크플로우

### Step 1: 기존 글 중복 체크

글을 쓰기 전에 반드시 기존 블로그 글 목록을 확인한다.

```bash
ls frontend/content/blog/*.mdx
```

각 파일의 frontmatter(title, category, tags)를 읽어서:
- 같은 주제의 글이 이미 있는지 확인
- 유사한 키워드를 다루는 글이 있는지 확인
- 중복이 있으면 사용자에게 알리고, 기존 글 보강 vs 신규 작성 중 선택하게 한다

### Step 2: 주제 기획

사용자가 주제를 지정하지 않은 경우, 기존 글 분석 후 빠진 주제를 제안한다.

**카테고리 체계:**
- `SQLD` — SQLD 관련 (공부법, 개념, 일정, 합격률 등)
- `정보처리기사` — 정처기 관련
- `컴퓨터활용능력` — 컴활 관련
- `일반` — 자격증 비교, CBT 활용법 등 공통 주제

**SEO 고려사항:**
- 검색량 높은 키워드를 title과 description에 포함
- title은 50자 이내, 핵심 키워드를 앞쪽에 배치
- description은 120자 이내, 검색 결과에 노출되는 문장
- tags에 검색 유입 키워드 5–7개 포함

### Step 3: 초안 작성

**파일 위치:** `frontend/content/blog/{slug}.mdx`

**frontmatter 형식:**
```yaml
---
title: "제목 — 부제목"
description: "검색 결과에 보이는 설명 (120자 이내)"
date: "YYYY-MM-DD"
category: "카테고리명"
tags: ["태그1", "태그2", ...]
---
```

**slug 규칙:**
- 영문 kebab-case: `sqld-study-guide`, `engineer-code-solving-tips`
- 카테고리 접두어 권장: `sqld-*`, `engineer-*`, `computer-literacy-*`

**문체 규칙:**
- 블로그 후기체 + 현실 조언체
- 존댓말(해요체) 기본
- AI 특유의 교과서식 문장 금지
- "혁신적인", "원활한", "차세대", "게임 체인저" 같은 AI 클리셰 금지
- 과한 단정 표현 완화 ("거의 필수" → "도움이 됩니다", "무조건" → "대부분의 경우")
- 독자가 자기 상황에 대입할 수 있는 구체적 예시 사용
- 표(table) 활용 후 반드시 해석 문단 추가 — 표만 보고 끝나면 안 됨

**컴포넌트 사용:**
- `<Highlight color="red|amber|green">텍스트</Highlight>` — 강조 박스
- `<PassRateBar />` — 합격률 차트 (합격률 글에서만)

**주의: 물결표(~) 사용 금지**
- `2~3주` → `2–3주` (en dash 사용)
- remark-gfm이 `~`를 취소선으로 해석하는 버그 방지

### Step 4: 자체 검증 루프

초안 작성 후 반드시 아래 체크리스트로 검증한다. 통과하지 못하면 수정 후 재검증.

**내용 검증:**
- [ ] 없는 사실이나 근거 없는 수치를 새로 만들지 않았는가
- [ ] AI스러운 과한 단정 표현이 없는가 ("무조건", "거의 필수", "가장 인정받는")
- [ ] 독자가 읽고 실제 판단/행동에 도움이 되는 글인가
- [ ] "맞아 나도 이거 고민했는데" 느낌이 나는가

**구조 검증:**
- [ ] 도입부가 평범하지 않고 문제 의식이나 궁금증을 유발하는가
- [ ] 표 뒤에 해석 문단이 있는가
- [ ] 마무리가 흔한 자기계발 문장이 아닌 실용적 정리인가
- [ ] 중간에 읽기를 멈출 만한 지루한 구간이 없는가

**기술 검증:**
- [ ] frontmatter 형식이 올바른가 (title, description, date, category, tags)
- [ ] 물결표(~)가 숫자 사이에 사용되지 않았는가 (en dash – 사용)
- [ ] Highlight 컴포넌트 문법이 올바른가
- [ ] slug가 kebab-case이고 기존 파일과 중복되지 않는가

**SEO 검증:**
- [ ] title에 핵심 검색 키워드가 포함되어 있는가
- [ ] description이 120자 이내이고 검색 결과에서 매력적인가
- [ ] tags가 5–7개이고 실제 검색 키워드인가
- [ ] h2/h3 헤딩에 검색 키워드가 자연스럽게 포함되어 있는가

### Step 5: 저장 및 커밋

검증 통과 후:
1. 파일을 `frontend/content/blog/{slug}.mdx`에 저장
2. `git add` + `git commit` + `git push`
3. 커밋 메시지: `feat(blog): {글 제목 요약}`

## 글 품질 기준

### 좋은 글의 특징
- 검색해서 들어온 취준생이 저장해두고 참고할 만한 수준
- 구체적인 상황별 추천이 있다 (개발 취업 / 공기업 / 비전공 등)
- 비유보다 실제 판단 기준을 제시한다
- 독자 유형별로 "나는 이렇게 하면 되겠구나"를 바로 알 수 있다

### 피해야 할 글의 특징
- 정보만 나열하고 판단을 도와주지 않는 글
- AI가 쓴 것처럼 매끈하지만 깊이가 없는 글
- 광고 문구처럼 보이는 마무리
- 교과서를 요약한 것 같은 글

## 트러블슈팅

### 빌드 에러: Module not found
- blog.ts의 getAllPosts()가 파싱 실패 → frontmatter YAML 형식 확인

### 취소선이 표시됨
- 물결표(~) 사용 금지. en dash(–)로 교체

### 이미지/썸네일 에러
- thumbnail 필드는 선택 사항. 없으면 생략
