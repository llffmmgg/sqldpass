# Progress Report 스킬

작업 완료 후 프로젝트의 구현 현황을 마크다운 문서로 자동 정리하는 스킬이다.

## 1. 문서 생성 전 분석 단계

문서를 작성하기 전에 반드시 다음을 수행한다:

- `git diff`와 `git log`로 최근 변경사항을 파악한다
- 프로젝트 디렉토리 구조를 스캔한다 (`ls`, `Glob` 등 활용)
- `package.json`, `build.gradle`, `requirements.txt` 등에서 기술 스택을 파악한다
- 기존 문서(`docs/PROGRESS.md`, `docs/CHANGELOG.md`, `docs/ARCHITECTURE.md`)가 있으면 읽어서 이전 내용을 유지한다

## 2. 생성할 문서 종류

사용자 요청에 따라 아래 중 선택하여 생성한다. 특정 문서를 지정하지 않으면 **PROGRESS.md를 기본**으로 생성한다. "전체 정리" 요청 시 세 문서를 모두 생성한다.

### a) `docs/PROGRESS.md` — 전체 구현 현황 (기본)

아래 템플릿을 따른다:

```markdown
# 프로젝트명 - 구현 현황
> 최종 업데이트: YYYY-MM-DD HH:MM

## 프로젝트 개요
한줄 설명

## 기술 스택
- Frontend: ...
- Backend: ...
- DB: ...
- Infra: ...

## 구현 완료
- [x] 기능명 - 간단한 설명 (관련 파일: `path/to/file`)

## 진행 중
- [ ] 기능명 - 현재 상태 설명

## TODO
- [ ] 기능명 - 설명

## 프로젝트 구조
주요 디렉토리/파일 트리

## API 엔드포인트 (백엔드인 경우)
| Method | Path | 설명 | 상태 |
|--------|------|------|------|

## 페이지/컴포넌트 (프론트엔드인 경우)
| 컴포넌트명 | 경로 | 설명 | 상태 |
|------------|------|------|------|
```

### b) `docs/CHANGELOG.md` — 날짜별 변경 이력

```markdown
# Changelog

## [YYYY-MM-DD]
### Added
- 새로 추가된 기능
### Changed
- 변경된 기능
### Fixed
- 수정된 버그
### Removed
- 제거된 기능
```

### c) `docs/ARCHITECTURE.md` — 아키텍처 문서

```markdown
# 아키텍처

## 시스템 구조
전체 구조 설명 + ASCII 다이어그램

## 디렉토리 구조
트리 구조 + 각 디렉토리 역할

## 데이터 흐름
요청-응답 흐름 설명

## 주요 모듈
각 모듈의 역할과 의존성
```

## 3. 작성 규칙

- **한국어**로 작성한다 (기술 용어는 영어 유지)
- `git log`에서 실제 변경된 파일 기반으로 작성한다 (추측 금지)
- 코드를 직접 읽어서 실제 구현된 기능만 "완료"로 표시한다
- 이전 문서가 있으면 덮어쓰지 말고 **업데이트**한다
- 날짜/시간은 항상 포함한다
- 체크리스트 형식으로 한눈에 진행률을 파악할 수 있게 작성한다

## 4. 기본 동작 요약

| 사용자 요청 | 생성 문서 |
|------------|----------|
| 특정 문서 미지정 | `docs/PROGRESS.md` |
| "전체 정리" | `docs/PROGRESS.md` + `docs/CHANGELOG.md` + `docs/ARCHITECTURE.md` |
| 특정 문서 지정 | 해당 문서만 |

`docs/` 폴더가 없으면 자동으로 생성한다.
